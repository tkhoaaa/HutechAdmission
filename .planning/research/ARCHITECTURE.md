### Structure Rationale

- **`public/locales/`**: Served as static assets, loaded at runtime by `i18next-http-backend`. Not bundled -- keeps JS chunks small. Vite serves these from `public/` automatically.
- **`src/i18n/`**: Init config, custom hooks, inline fallback strings. Separated from components for clear ownership.
- **`src/contexts/NotificationContext.jsx`**: Centralizes SSE connection management, FCM subscription state, and unread count. Components subscribe; context manages the lifecycle.
- **`backend/services/sseService.js`**: Singleton that holds connected `res` objects. Emit events when database records change (new application, status update).

## Architectural Patterns

### Pattern 1: PWA Service Worker via vite-plugin-pwa

**What:** Vite plugin that generates a Workbox-powered Service Worker during `vite build`, injected into `index.html` automatically.

**When to use:** Any PWA feature (offline, caching, push notifications).

**Trade-offs:** The plugin auto-generates `sw.js` in the build output (`dist/`). You configure strategies in `vite.config.js`. During dev, it can use a devtools-friendly approach (not full SW). After build, the SW is served from the Vercel static host.

**Integration with existing `manifest.json`:**
- The existing `public/manifest.json` is referenced by `index.html` via `<link rel="manifest">`.
- vite-plugin-pwa can auto-generate a manifest or you can reference the existing one via `injectManifest: false` and `manifest: false` config.
- **Recommendation:** Keep existing `manifest.json`, configure vite-plugin-pwa to reference it via `srcDir`/`filename` or simply add the `id: "sw-base"` field to existing manifest.

**Example vite.config.js addition:**
```javascript
import { VitePWA } from 'vite-plugin-pwa'

export default defineConfig({
    plugins: [
        react(),
        VitePWA({
            registerType: 'autoUpdate',
            injectRegister: 'auto',
            workbox: {
                globPatterns: ['**/*.{js,css,html,ico,png,svg,woff2}'],
                runtimeCaching: [
                    {
                        urlPattern: /^https:\/\/do-an-tuyen-sinh\.vercel\.app\/api\//,
                        handler: 'NetworkFirst',
                        options: {
                            cacheName: 'api-cache',
                            networkTimeoutSeconds: 5,
                            expiration: { maxEntries: 50, maxAgeSeconds: 86400 }
                        }
                    },
                    {
                        urlPattern: /^https:\/\/fonts\.(googleapis|gstatic)\.com/,
                        handler: 'CacheFirst',
                        options: {
                            cacheName: 'google-fonts',
                            expiration: { maxEntries: 10, maxAgeSeconds: 2592000 }
                        }
                    }
                ]
            },
            manifest: false  // Use existing manifest.json
        })
    ]
})
```

### Pattern 2: SSE Endpoint in Express

**What:** HTTP endpoint that keeps the response connection open and streams events to the client. Unlike WebSocket, SSE is unidirectional (server-to-client only) and works over standard HTTP.

**When to use:** Admin gets live updates on new applications; candidates get status change notifications. No client-to-server messaging needed.

**Trade-offs:** SSE connections are long-lived -- the Express server must manage a registry of connected clients. Each connected client consumes a server thread/connection. For low-to-medium traffic (this project's scale), this is fine. Vercel serverless functions do NOT support persistent SSE connections (functions are stateless), so SSE only works against the backend server on port 3001, not Vercel.

**Critical integration point:** The CORS config in `backend/index.js` currently allows `localhost:5173` and `do-an-tuyen-sinh.vercel.app`. The SSE endpoint must share this CORS setup.

**Backend SSE implementation:**
```javascript
// backend/services/sseService.js
class SSEService {
    constructor() {
        this.clients = new Map(); // userId -> Set<res>
    }

    addClient(userId, res) {
        if (!this.clients.has(userId)) this.clients.set(userId, new Set());
        this.clients.get(userId).add(res);
    }

    removeClient(userId, res) {
        if (this.clients.has(userId)) {
            this.clients.get(userId).delete(res);
            if (this.clients.get(userId).size === 0) this.clients.delete(userId);
        }
    }

    broadcast(userId, event, data) {
        const clientSet = this.clients.get(userId);
        if (!clientSet) return;
        const message = `event: ${event}\ndata: ${JSON.stringify(data)}\n\n`;
        for (const res of clientSet) {
            res.write(message);
        }
    }

    broadcastAll(event, data) {
        const message = `event: ${event}\ndata: ${JSON.stringify(data)}\n\n`;
        for (const [, clientSet] of this.clients) {
            for (const res of clientSet) res.write(message);
        }
    }
}

export default new SSEService();

// backend/index.js -- SSE endpoint
import sseService from './services/sseService.js';

app.get('/api/sse/events', authenticateToken, (req, res) => {
    res.setHeader('Content-Type', 'text/event-stream');
    res.setHeader('Cache-Control', 'no-cache');
    res.setHeader('Connection', 'keep-alive');
    res.setHeader('X-Accel-Buffering', 'no');  // Disable nginx buffering
    res.flushHeaders();

    const userId = req.user.id;
    const isAdmin = req.user.role === 'admin';

    sseService.addClient(isAdmin ? 'admin' : userId, res);

    // Send heartbeat every 30s to keep connection alive
    const heartbeat = setInterval(() => {
        res.write(': heartbeat\n\n');
    }, 30000);

    req.on('close', () => {
        clearInterval(heartbeat);
        sseService.removeClient(isAdmin ? 'admin' : userId, res);
    });
});

// Trigger SSE from existing routes when data changes
// e.g., after new application POST succeeds:
app.post('/api/applications', ..., async (req, res) => {
    // ... existing logic ...
    // After save:
    sseService.broadcast('admin', 'new_application', {
        id: newApp.id,
        candidateName: newApp.fullName,
        timestamp: new Date().toISOString()
    });
    // For candidate:
    sseService.broadcast(userId, 'status_update', {
        applicationId: appId,
        status: newStatus
    });
});
```

### Pattern 3: i18n with Lazy-Loaded Locale Files

**What:** Translation strings stored in `public/locales/{lang}/{namespace}.json`, loaded asynchronously by `i18next-http-backend` only when needed.

**When to use:** Multi-language support with bundle size constraint (50KB max increase).

**Trade-offs:** First language switch has a network request (JSON file). Subsequent switches may use browser cache. Fallback language loads synchronously as inline JS to prevent flash-of-untranslated-content (FOTUC).

**Why `public/locales/` instead of `src/locales/`:**
- Files in `public/` are served as static assets, not bundled by Vite.
- `i18next-http-backend` fetches them via HTTP at runtime.
- This keeps JS bundle small -- only the i18n library code (~23KB gzipped for full react-i18next) is in the bundle.
- Locale JSON files are cached by the browser.

**i18n config (`src/i18n/config.js`):**
```javascript
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import HttpBackend from 'i18next-http-backend';
import LanguageDetector from 'i18next-browser-languagedetector';

i18n
    .use(HttpBackend)           // Load from /locales/{lng}/{ns}.json
    .use(LanguageDetector)     // Detect browser language
    .use(initReactI18next)
    .init({
        fallbackLng: 'vi',
        defaultNS: 'translation',
        ns: ['translation', 'common'],
        backend: {
            loadPath: '/locales/{{lng}}/{{ns}}.json'
        },
        detection: {
            order: ['localStorage', 'navigator'],
            caches: ['localStorage']
        },
        interpolation: {
            escapeValue: false  // React already escapes
        }
    });

export default i18n;
```

**LanguageSwitcher integration into existing header:**
The existing `ThanhHeader.jsx` renders the navigation. Add `LanguageSwitcher` component into the header's right section (near ThemeToggle). It reads/writes the language to localStorage via i18next detector, triggering a re-render of all translated strings.

## Anti-Patterns to Avoid

### Anti-Pattern 1: Loading All Locales in the JS Bundle

**What people do:** Import all translation JSON files as modules (`import vi from './locales/vi.json'`) and pass them to i18next init.

**Why it's wrong:** All translation strings become part of the JavaScript bundle. With two languages and multiple namespaces, this adds 30-100KB+ to every chunk that imports i18n. Violates the 50KB bundle budget.

**Do this instead:** Store locale files in `public/locales/` and use `i18next-http-backend` to fetch them at runtime. Only the i18n library (~23KB gzipped) goes into the bundle.

### Anti-Pattern 2: SSE Without Heartbeat

**What people do:** Open an SSE connection without periodic heartbeat messages.

**Why it's wrong:** Proxies, load balancers, and some corporate firewalls close long-lived connections that appear idle. Without heartbeat, the connection dies silently after ~30-60 seconds.

**Do this instead:** Send a comment heartbeat (`res.write(': ping\n\n')`) every 30 seconds. The client should reconnect on error. The `NotificationContext` should handle reconnection with exponential backoff.

### Anti-Pattern 3: SW Caching All API Requests

**What people do:** Use `CacheFirst` strategy for all API requests, so stale data is served even when online.

**Why it's wrong:** Admissions data, application status, and personal records must be current. Serving stale cached data creates confusion and incorrect decisions.

**Do this instead:** Use `NetworkFirst` with a short timeout (3-5 seconds) for API requests. If network fails, fall back to cache. For admin pages with real-time needs, consider skipping SW cache entirely for those routes.

### Anti-Pattern 4: Not Wrapping i18n with Suspense

**What people do:** Initialize i18next with async backend loading but render translated content before i18next is ready.

**Why it's wrong:** On first load, locale files must be fetched. Without Suspense, translated content shows as empty keys (e.g., `translation.key`) briefly.

**Do this instead:** In `main.jsx`, wrap the App with `<Suspense fallback={<PageLoader />}>`. The `PageLoader` component already exists and is used for page-level lazy loading -- reuse it for i18n hydration.

### Anti-Pattern 5: SSE on Vercel Serverless

**What people do:** Deploying SSE endpoint as a Vercel serverless function.

**Why it's wrong:** Serverless functions are stateless and short-lived (max ~10s for Hobby, ~30s for Pro). SSE requires a persistent connection that outlives function invocations. Vercel will terminate the connection immediately.

**Do this instead:** SSE must connect to the backend Express server on port 3001 (self-hosted or VPS). The Vercel-hosted frontend proxies SSE requests to the backend, or the backend domain is used directly. Vercel edge functions also do not support SSE.

## Scaling Considerations

| Scale | Architecture Adjustments |
|-------|--------------------------|
| 0-100 concurrent users | SSE with single Express server is fine. No changes needed. |
| 100-500 concurrent users | SSE connection registry may grow. Monitor memory. Consider Redis pub/sub if scaling to multiple backend instances. |
| 500+ concurrent users | SSE becomes a bottleneck. Migrate to a message broker (Redis/Socket.io) or use a dedicated real-time service (Pusher/Ably). |

### Scaling Priorities

1. **First bottleneck:** SSE connection memory on Express server. Each connection holds a `res` object. With 1000 concurrent users, ~1-2MB of memory. Manageable but monitor.
2. **Second bottleneck:** SSE + MySQL on same server. Consider read replicas for MySQL if SSE broadcasts cause CPU spikes.
